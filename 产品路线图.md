# IEC104 协议测试工具 产品化路线

## 一、现状评估

### 已有优势

- 客户端/服务端双模式，覆盖测试主链路
- 多连接管理，适合复杂测试场景
- 报文捕获、解析、存储、导出功能完整
- 基于 j60870 成熟协议栈

### 当前短板

- 仅支持 IEC 104 单一协议
- 缺少自动化测试能力（全手动操作）
- 无测试报告生成
- Swing 界面偏老旧，缺乏专业感
- 无安装包，用户需自行配 Java 环境
- 无单元测试和 CI/CD 流程

## 二、产品定位

| 维度 | 定位 |
|------|------|
| 目标用户 | 电力二次设备厂商、系统集成商、电力调试工程师、SCADA 开发团队 |
| 核心价值 | 让 IEC 104 协议测试从"手工抓包"升级为"标准化测试" |
| 竞品参考 | Triangle MicroWorks Probe、KEMA IEC104Test、Wireshark（被动抓包） |
| 差异化 | 轻量、国产化、中文友好、可脚本化自动化测试 |

## 三、版本路线图

### Phase 1：产品基础打磨（v1.1 ~ v1.3）

**目标：让工具从"能用"变成"好用"**

#### v1.1 — 体验优化

- 界面美化：统一图标、配色方案、间距规范，提升专业感
- 报文列表增加过滤、搜索、高亮功能（按类型、方向、IOA 筛选）
- 报文详情面板增加十六进制原始数据视图
- 连接配置保存与加载（JSON 格式，可复用）
- 常用操作快捷键支持

#### v1.2 — 稳定性与打包

- 引入 JUnit 单元测试，覆盖 ASDU 编解码核心逻辑
- 异常处理完善（网络断开、协议异常的优雅处理）
- 使用 jpackage 打包为免 JDK 安装的可执行程序（.exe / .dmg）
- 首次启动引导和帮助文档
- 版本更新检查机制

#### v1.3 — 测试场景化

- 场景保存：一套连接配置 + 数据点 + 服务端配置，一键加载
- 批量测试用例管理：定义"发送命令→预期响应"的用例
- 测试日志记录：自动记录操作时间线，便于追溯
- 简单脚本支持（如 Lua 或 JavaScript 引擎嵌入）

### Phase 2：功能纵深（v2.0 ~ v2.5）

**目标：从"测试工具"升级为"测试平台"**

#### v2.0 — 自动化测试引擎

- 脚本引擎：支持编写自动化测试脚本（Nashorn/GraalJS）
- 测试套件：用例编排、批量执行、断言校验
- 定时/循环测试：压力测试、长时间稳定性测试
- 测试报告自动生成（HTML/PDF，含通过率、异常截图）

#### v2.1 — 协议扩展

- IEC 60870-5-103（串口/网络）
- IEC 60870-5-101（串口）
- Modbus TCP/RTU
- 多协议并行测试，统一界面

#### v2.2 — 数据与分析

- SQLite 存储测试历史，支持查询和对比
- 实时数据曲线图（遥测值趋势）
- 报文统计仪表盘（成功率、响应时间、异常分布）
- 多次测试结果对比分析

#### v2.5 — 协议合规性测试

- 标准化测试套件：按 IEC 60870-5-104 标准条款设计
- 互操作性测试场景库
- 协议一致性报告（对标国家电网/南网检测要求）

### Phase 3：产品化与商业化（v3.0+）

**目标：建立商业壁垒和生态**

#### v3.0 — 架构升级

- 插件化架构：协议模块、测试模块、报表模块均可插拔
- 后端服务化：核心引擎与界面分离，支持 CLI 和 API 调用
- 远程协作：多终端共享测试会话（C/S 架构）

#### v3.1 — 企业级特性

- 用户权限管理
- 测试资产统一管理（配置、脚本、报告）
- CI/CD 集成（提供 CLI 和 REST API，可接入 Jenkins 等）
- 设备台账管理（关联测试目标设备信息）

#### v3.2 — 商业化

- 免费版 + 专业版 + 企业版分级授权
- 免费版：基础客户端/服务端、报文解析
- 专业版：自动化测试、报告生成、多协议
- 企业版：团队协作、API 集成、定制开发

## 四、技术架构演进

### 架构对比

```
当前架构（v1.0）               目标架构（v3.0）
┌─────────────┐               ┌──────────────────────────┐
│   Swing UI  │               │  Web UI / JavaFX UI      │
│      ↕      │               │      ↕  REST/gRPC        │
│  Core Logic │    ──────►    │  Test Engine (核心服务)   │
│  (耦合)     │               │  ├─ Protocol Plugins     │
│      ↕      │               │  ├─ Script Engine        │
│   j60870    │               │  ├─ Report Generator     │
└─────────────┘               │  └─ Data Store (SQLite)  │
                              └──────────────────────────┘
```

### 短期建议（不改架构）

- 梳理代码，消除 `j60870/` 根目录与 `src/` 下的重复源码
- 引入简单的依赖注入，降低 UI 与 Core 的耦合
- 建立分层：`ui` → `service` → `core` → `protocol`

## 五、近期优先事项

1. **清理项目结构**：删除重复的 j60870 源码，统一到 Maven 标准结构
2. **打包成 exe**：用 jpackage 让用户免装 JDK 即可运行
3. **报文过滤搜索**：这是日常调试最高频的需求
4. **场景配置保存**：让用户不用每次重新配置
5. **建立测试用例**：为 ASDU 编解码写单元测试，保证后续迭代不引入 bug

## 六、风险提示

| 风险 | 应对 |
|------|------|
| j60870 库维护停滞 | 评估自研协议栈的可能性，或 fork 维护 |
| Swing 技术栈老化 | v2.0 起评估迁移到 JavaFX 或 Web 前端 |
| 协议标准复杂度高 | 优先覆盖国标常见场景，逐步完善 |
| 商业化竞争 | 聚焦国产化和中文场景，做差异化 |
